# JavaScript Clean Code Guide for 쑥쑥뽑아

이 문서는 `clean-code-javascript-kr`의 핵심 원칙을 **쑥쑥뽑아** 같은 작은 브라우저 JavaScript 앱에 맞게 축약·적용한 프로젝트용 가이드다.

이 문서는 스타일 가이드가 아니라, **읽기 쉽고, 수정하기 쉽고, 동작 보존이 쉬운 코드**를 만들기 위한 실천 기준이다.
강박적으로 전부 적용하지 말고, **작은 앱의 단순함을 해치지 않는 선에서** 사용한다.

---

## 1. 이 프로젝트에서 가장 중요한 원칙

### 1.1 상태를 믿고, DOM을 다시 해석하지 않는다

좋은 예:
- `state.currentBall`
- `state.balls`
- `state.pickedBalls`

나쁜 예:
- 이미 상태에 있는 값을 `innerText`, `id`, DOM 개수로 다시 읽어서 상태처럼 쓰는 것

규칙:
- 상태 변경은 `state`에 반영한다.
- 렌더링은 상태를 기반으로 수행한다.
- DOM은 결과물이지, 진실 원천이 아니다.

### 1.2 전역 결합을 줄인다

좋은 예:
- `picking(state, elements)`
- `bindSetupModal(state, storage, firstN, elements)`

나쁜 예:
- 다른 파일의 전역 변수 이름을 직접 참조하는 것

규칙:
- 필요한 것은 함수 인자로 받는다.
- 숨은 의존성보다 명시적 의존성을 선호한다.

### 1.3 동작 보존이 리팩토링보다 우선이다

규칙:
- 리팩토링과 기능 변경을 섞지 않는다.
- 사용자 체감 동작이 달라지면 별도 변경으로 취급한다.

---

## 2. 변수 규칙

### 2.1 의도가 보이는 이름을 쓴다

좋은 예:
- `currentBall`
- `pickedBalls`
- `parseExcludedNumbers`
- `renderPickedBalls`

피해야 할 예:
- `arr`, `tmp`, `n`, `data2`

예외:
- 짧은 반복문의 `i` 정도는 허용 가능
- 다만 의미 있는 이름이 훨씬 읽기 쉬우면 더 긴 이름을 쓴다

### 2.2 같은 개념에는 같은 이름을 쓴다

규칙:
- `pickedBalls`를 어떤 곳에서는 `selectedBalls`, 어떤 곳에서는 `historyBalls`처럼 바꾸지 않는다.
- 같은 역할의 DOM 참조는 `elements` 객체 안에서 일관되게 관리한다.

### 2.3 매직 넘버를 의도 있는 이름으로 바꾼다

좋은 예:
- `const firstN = 10`

추가 후보:
- 추첨 interval의 `60`
- 화면 폭 기준값 `550`, `450`, `380`, `300`

이 숫자들은 반복해서 의미를 가지면 상수로 올린다.

---

## 3. 함수 규칙

### 3.1 함수는 한 가지 일에 집중한다

좋은 분리 예:
- `parseExcludedNumbers()` → 입력 파싱
- `renderPickedBalls()` → 하단 공 목록 렌더
- `clearCurrentBall()` → 중앙 공 상태/표시 초기화

피해야 할 예:
- 입력 검증 + 상태 변경 + 렌더링 + 저장을 한 함수에서 모두 처리

### 3.2 함수 이름은 동사를 포함해 의도를 드러낸다

좋은 예:
- `initBalls`
- `moveBalls`
- `renderPickedBalls`
- `bindSetupModal`
- `clearCurrentBall`

### 3.3 인자가 많아지면 묶을 수 있는지 본다

현재 이 프로젝트에서는 다음 패턴을 기본으로 한다.

- 상태 관련: `state`
- DOM 관련: `elements`
- 저장소 관련: `storage`

규칙:
- 같은 종류의 의존성은 객체로 묶는다.
- 하지만 작은 앱에 클래스/과한 추상화를 억지로 넣지는 않는다.

### 3.4 불리언 플래그 인자는 피한다

나쁜 예:
- `renderBall(true)`
- `updateState(false)`

좋은 예:
- `renderPickedBalls()`
- `clearCurrentBall()`
- `startPicking()` / `stopPicking()`처럼 분리 가능한지 검토

### 3.5 가드 절을 우선 고려한다

좋은 예:
- `if (!target) return`
- `if (!state.currentBall) return`

중첩을 줄이고 실패 조건을 빨리 드러낸다.

---

## 4. 상태 관리 규칙

### 4.1 상태 초기화는 한 곳에 모은다

규칙:
- 타이머 정지
- `stopped`
- `currentBall`
- `balls`
- `pickedBalls`
- 중앙 UI 정리

이런 초기화는 흩어놓지 말고, 가능한 한 중앙 함수에서 처리한다.

### 4.2 상태 전환은 예측 가능해야 한다

예:
- picking 시작 시
  - `stopped = false`
  - timer 시작
- picking 종료 시
  - `stopped = true`
  - timer 종료
  - 공 확정 처리

규칙:
- 상태 전환이 일어날 때 어떤 필드가 바뀌는지 읽는 사람이 추론 가능해야 한다.

### 4.3 파생 정보는 필요 이상으로 상태에 저장하지 않는다

규칙:
- 상태에서 계산 가능한 값은 매번 계산해도 된다.
- 하지만 DOM에서 다시 읽어오는 것보다는 상태/헬퍼 계산이 낫다.

---

## 5. 렌더링 규칙

### 5.1 렌더링은 상태에서 직접 만든다

좋은 예:
- `renderPickedBalls(state, elements)`가 `state.pickedBalls`를 기준으로 다시 그림

나쁜 예:
- DOM의 `.picked-ball`을 다시 읽어 배열을 만든 뒤 재렌더링

### 5.2 렌더링 함수는 이름만 보고도 무엇을 그리는지 알 수 있어야 한다

좋은 예:
- `renderPickedBalls`
- `clearCurrentBall`
- `emptyContainer`

### 5.3 DOM 쿼리는 필요한 범위 안으로 제한한다

좋은 예:
- `elements.ballContainerDiv.querySelectorAll(...)`

나쁜 예:
- 앱 어디서든 전역 `document.querySelectorAll(...)`를 무분별하게 사용하는 것

---

## 6. 입력 처리 규칙

### 6.1 입력값은 저장 전에 정규화한다

예:
- 공백 제거
- 숫자 변환
- 범위 검사
- 중복 제거

좋은 예:
- `parseExcludedNumbers(value, total)`

### 6.2 검증 로직과 파싱 로직을 분리한다

가능하면 다음 순서를 따른다.

1. 문자열 형식 검사
2. 파싱
3. 정규화
4. 저장
5. 렌더/초기화

---

## 7. 주석 규칙

### 7.1 주석은 “무엇”보다 “왜”를 설명한다

좋은 예:
- 왜 `closest(".picked-ball")`를 쓰는지
- 왜 마지막 공 처리에서 `currentBall`을 먼저 세팅하는지

나쁜 예:
- 코드가 이미 말하는 내용을 그대로 반복하는 주석

### 7.2 죽은 코드 주석은 지운다

예:
- 오래된 대안 구현
- 이미 버린 실험 코드
- 지금은 쓰지 않는 대량 주석 블록

규칙:
- 필요하면 Git 히스토리로 찾고, 현재 파일은 가볍게 유지한다.

---

## 8. 중복 제거 규칙

### 8.1 반복되는 상태/UI 정리는 함수로 모은다

좋은 후보:
- `clearCurrentBall`
- `renderPickedBalls`
- `emptyContainer`
- `parseExcludedNumbers`

### 8.2 단, 한 번만 쓰는 작은 코드까지 억지로 함수로 빼지 않는다

규칙:
- 중복 제거보다 읽기 쉬움이 더 중요하다.
- 함수 추출이 오히려 흐름 이해를 어렵게 만들면 유지한다.

---

## 9. 이 프로젝트에서 피할 것

- 다른 파일 전역변수 이름에 직접 의존하기
- DOM을 상태처럼 다시 읽기
- 리팩토링과 기능 변경 섞기
- 짧지만 의미 없는 이름 쓰기
- 한 함수에 검증, 저장, 상태 변경, 렌더링을 모두 몰아넣기
- 필요 이상의 추상화 도입
- 프레임워크/모듈 시스템을 리팩토링 명목으로 끼워 넣기

---

## 10. 권장 리팩토링 순서

큰 리팩토링을 할 때는 보통 아래 순서를 따른다.

1. 현재 동작을 짧게 기록
2. 상태 필드와 렌더링 기준을 명확히 정리
3. 이름 개선
4. 작은 함수 분리
5. 중복 제거
6. 숨은 결합 제거
7. 수동 스모크 체크

---

## 11. 수동 체크리스트

리팩토링 후 최소한 아래는 확인한다.

- 첫 로드가 정상인지
- `시작` → `뽑아` → `쑥쑥` 흐름이 정상인지
- 마지막 공을 뽑은 뒤 `다시`가 동작하는지
- 설정 저장/적용이 되는지
- 제외 번호가 반영되는지
- 뽑힌 공을 클릭하면 다시 풀로 돌아가는지
- 초기화가 타이머, 중앙 공, 하단 공 목록까지 정리하는지

---

## 12. 적용 원칙 요약

이 프로젝트에서 클린코드는 아래 문장으로 요약한다.

- 상태는 명시적으로 관리한다.
- 렌더링은 상태에서 만든다.
- 함수는 한 가지 책임에 집중한다.
- 이름은 의도를 드러낸다.
- 리팩토링은 동작 보존을 전제로 작게 나눈다.
