# Display / Control 분리 스펙 초안

## 목적

현재 `쑥쑥뽑아`에는 진행제어 카드(예약 기능)가 이미 구현되어 있으며, 현재는 기존 메인 화면인 `index.html`에서도 해당 UI를 함께 볼 수 있다.

하지만 실제 사용 목적상 진행제어 기능은 **선생님만 사용하는 관리용 기능**이어야 하며, 학생들에게 보여지는 발표/추첨 화면에는 노출되지 않아야 한다.

이번 작업의 목표는 기존 메인 화면의 동작을 크게 바꾸지 않으면서,

- 기존 메인 **display 페이지(index.html)**
- 선생님용 **control 페이지(control.html)**

를 역할에 맞게 분리하는 것이다.

---

## 현재 전제

- 진행제어 카드(예약 기능)는 이미 동작하는 기능으로 간주한다.
- 현재 정책은 아래와 같이 본다.
  - 예약은 **다음 1회만** 적용된다.
  - 예약된 번호는 **다음 결과를 100% 고정**한다.
  - 예약은 **1장만 유지**한다.
  - 새로고침 후에는 예약이 유지되어도 괜찮다.
  - 초기화 또는 설정 변경 시에는 예약도 함께 초기화한다.
- 이번 단계의 핵심은 진행제어 기능 자체를 확장하는 것이 아니라,
  **이 기능을 어느 화면에서 보여줄 것인지 분리하는 것**이다.

---

## 이번 작업의 핵심 목표

1. `index.html`은 기존 메인 화면 역할을 유지한다.
2. 관리용 진행제어 UI는 별도 `control.html` 페이지로 분리한다.
3. 두 화면은 가능한 한 같은 앱 상태를 공유하도록 유지한다.
4. 기존처럼 브라우저에서 `index.html`을 직접 열어도 메인 화면이 동작해야 한다.
5. 기존 추첨 흐름(시작/정지/마지막 공/다시/설정/되돌리기)은 `index.html`에서 최대한 그대로 유지한다.

---

## 역할 분리 방향

### 1. display 페이지

학생들에게 보여지는 메인 화면.
이 문서에서 display 페이지는 새 파일이 아니라 기존 `index.html`을 뜻한다.

포함 대상:
- 현재 뽑히는 번호 표시
- 이미 뽑힌 번호 목록
- 추첨 진행 결과 시각화
- 시작/정지 버튼
- 설정 버튼 및 설정 모달
- 클릭한 뽑힌 공 되돌리기

원칙:
- 이번 단계에서는 `index.html`의 기존 사용자 경험을 우선 보존한다.
- 진행제어 카드가 최종적으로 display에서 제거될 수는 있어도, 이번 단계의 최우선 목표는 기존 메인 화면을 깨지 않는 것이다.
- 따라서 구현 순서는 “control 페이지 추가”가 먼저이고, 기존 `index.html` 정리는 그 다음 단계로 미룰 수 있다.

즉, display는 기존 메인 앱을 유지하는 화면이다.

---

### 2. control 페이지

선생님이 조작하는 관리 화면.

포함 대상:
- 진행제어 카드(예약 입력/취소/상태/메시지)

제외 대상:
- 추첨 시작/정지 버튼
- 설정 버튼/설정 모달
- 메인 추첨 화면의 상호작용
- 뽑힌 공 클릭 되돌리기

즉, control 페이지는 **관리자용 예약 제어**를 담당한다.

---

## 구현 원칙

- 기존 동작을 크게 흔드는 전면 재작성은 피한다.
- 먼저 **최소 분리 구조**를 제안하고, 작은 단계로 나누어 진행한다.
- state를 소스 오브 트루스로 유지한다.
- DOM 표시 여부만으로 상태를 판단하지 않는다.
- 진행제어 로직 자체는 재작성하지 말고, 우선은 **노출 범위 분리**에 집중한다.
- control 전용 DOM과 display 전용 DOM의 책임을 명확히 나눈다.
- `index.html`의 기존 기능은 가능한 한 유지하고, 새 `control.html`을 추가하는 방식으로 시작한다.
- 이번 단계에서는 “display 전용 새 페이지 추가”보다 “control 전용 페이지 추가”를 우선한다.

---

## Codex에게 기대하는 작업 방식

이번 작업은 곧바로 큰 수정에 들어가기보다, 먼저 아래 순서로 진행한다.

1. 현재 구조에서 기존 `index.html`을 유지하면서 `control.html`을 추가하는 최소 설계를 정리한다.
2. 어떤 DOM 요소가 기존 display(index) 전용인지, 어떤 DOM 요소가 control 전용인지 식별한다.
3. 어떤 상태가 두 페이지에서 공유되어야 하는지 정리한다.
4. 가능한 가장 작은 구현 slice 1개를 제안한다.
5. 실제 코드 수정 전에는 반드시 diff 또는 patch summary를 먼저 제시한다.
6. 사용자 승인 전에는 저장소 파일을 수정하지 않는다.

---

## 이번 범위에 포함되는 것

- 기존 `index.html` 유지 전제 하의 display/control 역할 분리 설계
- 진행제어 UI를 별도 `control.html` 페이지로 분리하는 방향 정리
- 기존 구조에서 이를 가장 작게 적용할 수 있는 방법 제안
- 필요한 경우 공통 로직 분리 또는 control 페이지 추가 제안

---

## 이번 범위에 포함되지 않는 것

- 진행제어 카드의 다중 예약 기능
- 예약 정책 변경(확률 유도형으로 변경 등)
- 대규모 파일 구조 재작성
- 프레임워크/빌드 도구 도입
- 학생/교사 인증 시스템 도입

---

## 수동 확인 항목

의미 있는 변경 후에는 최소한 아래를 수동 확인 대상으로 제시한다.

1. `index.html`에서 추첨 시작/정지 흐름이 기존과 같게 동작하는지
2. `index.html`에서 설정 모달과 `다시`, 마지막 공 처리, 되돌리기 기능이 유지되는지
3. `control.html`에서 예약 입력/취소/상태 표시가 정상 동작하는지
4. `control.html`에서 입력한 예약이 `index.html`의 다음 추첨에 반영되는지
5. 설정 적용 후 예약 상태가 의도대로 초기화되는지
6. 새로고침 후 예약 유지 정책이 의도대로 동작하는지
7. `index.html`의 기존 UX가 불필요하게 바뀌지 않았는지

---

## 요청 사항

Codex는 먼저 `AGENTS.md`를 읽고 프로젝트 제약을 확인한 뒤,
이번 문서를 기준으로 작업을 진행한다.

이번 단계에서는 구현보다 먼저,

- 최소 설계안
- 작은 구현 단계
- 예상 diff summary

를 먼저 제시해야 한다.
